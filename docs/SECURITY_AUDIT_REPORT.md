# Отчет об аудите безопасности и функциональности

Дата: 2026-03-08

## Резюме

Проведен полный аудит безопасности и функциональности приложения Telegram Seller MVP. Найдено и исправлено **7 критических проблем безопасности** и несколько улучшений в бизнес-логике.

## Критические проблемы безопасности (ИСПРАВЛЕНО)

### 1. ⚠️ КРИТИЧНО: Небезопасный поиск магазина при авторизации

**Проблема:** В `auth/telegram/route.ts` и `auth/dev/route.ts` использовался запрос `shopRepo.findOne({ where: {} })`, который находил ЛЮБОЙ магазин в системе, не обязательно принадлежащий пользователю.

**Риск:** Пользователь мог получить доступ к чужому магазину.

**Исправление:** 
- Изменена логика поиска магазина через связь `UserShop`
- Магазин ищется только через `userId` пользователя
- Если магазина нет, создается новый для пользователя

**Файлы:**
- `src/app/api/auth/telegram/route.ts`
- `src/app/api/auth/dev/route.ts`
- `src/app/api/auth/key/route.ts`

### 2. ⚠️ КРИТИЧНО: Отсутствие подписи сессии

**Проблема:** Сессия хранилась в cookie как простой JSON без подписи, что позволяло пользователю подделать cookie и получить доступ к чужому аккаунту.

**Риск:** Подделка сессии и несанкционированный доступ.

**Исправление:**
- Добавлена функция подписи сессии с использованием HMAC-SHA256
- Сессия теперь хранится в формате `{data}:{signature}`
- Добавлена проверка подписи при чтении сессии
- Используется безопасное сравнение для предотвращения timing attacks

**Файлы:**
- `src/lib/auth.ts` - добавлены функции `signSession()`, `verifySession()`, `createSignedSession()`
- Все auth endpoints обновлены для использования подписанных сессий

**Требуется:** Добавить переменную окружения `SESSION_SECRET` в `.env`:
```env
SESSION_SECRET=ваш-секретный-ключ-минимум-32-символа
```

### 3. ⚠️ КРИТИЧНО: Отсутствие проверки принадлежности flavorId к магазину

**Проблема:** В некоторых местах при работе с `flavorId` не проверялось, что товар принадлежит правильному магазину.

**Риск:** Пользователь мог работать с товарами из чужого магазина.

**Исправление:**
- Добавлена проверка `shopId` при работе с `flavorId` в:
  - `src/app/api/sales/route.ts` - при создании продажи
  - `src/app/api/sales/[id]/route.ts` - при редактировании продажи
  - `src/app/api/inventory/stock/route.ts` - при обновлении остатков

**Файлы:**
- `src/app/api/sales/route.ts`
- `src/app/api/sales/[id]/route.ts`
- `src/app/api/inventory/stock/route.ts`

### 4. Проблема с оплатой долга

**Проблема:** Можно было принять оплату больше остатка долга, что приводило к некорректным данным.

**Исправление:** Добавлена проверка, что сумма оплаты не превышает остаток долга.

**Файл:** `src/app/api/debts/route.ts`

## Улучшения бизнес-логики

### 5. Улучшена валидация при создании остатков

**Проблема:** При создании нового `StockItem` не проверялось, что `flavorId` принадлежит правильному магазину.

**Исправление:** Добавлена проверка перед созданием остатка.

**Файл:** `src/app/api/inventory/stock/route.ts`

### 6. Улучшена обработка ошибок при валидации товаров

**Проблема:** При недостатке товара не всегда проверялась принадлежность к магазину.

**Исправление:** Добавлена явная проверка перед проверкой остатков.

**Файлы:**
- `src/app/api/sales/route.ts`
- `src/app/api/sales/[id]/route.ts`

## Рекомендации

### 1. Добавить переменную окружения SESSION_SECRET

Создайте файл `.env` (если его нет) и добавьте:
```env
SESSION_SECRET=ваш-секретный-ключ-минимум-32-символа-случайные-символы
```

**Важно:** В production используйте длинный случайный ключ!

### 2. Рекомендуется добавить rate limiting

Для защиты от брутфорса рекомендуется добавить rate limiting на:
- `/api/auth/key` - вход по ключу
- `/api/auth/telegram` - вход через Telegram

### 3. Рекомендуется добавить логирование

Добавить логирование:
- Неудачных попыток авторизации
- Попыток доступа к чужим данным
- Административных действий

### 4. Рекомендуется добавить CSRF защиту

Для дополнительной защиты можно добавить CSRF токены для критических операций (удаление, изменение данных).

После применения исправлений рекомендуется протестировать:

1. **Авторизацию:**
   - Вход через Telegram initData
   - Вход через access key

   - Оплата долгов с проверкой суммы
