# Чек-лист подготовки к Production

Этот документ содержит чек-лист для проверки готовности приложения к коммерческому запуску.

## Перед запуском

### 1. Переменные окружения

- [ ] `NODE_ENV=production` установлен
- [ ] `SESSION_SECRET` установлен (минимум 32 символа, уникальный для production)
- [ ] `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME` настроены для production БД
- [ ] `TELEGRAM_BOT_TOKEN` установлен (токен от BotFather)
- [ ] При использовании **webhook**: `TELEGRAM_WEBHOOK_URL` и `TELEGRAM_WEBHOOK_SECRET` заданы. При **long polling** (PM2 `telegram-bot-polling`) webhook не обязателен
- [ ] Все dev-only переменные удалены или отключены

### 2. База данных

- [ ] PostgreSQL установлен и запущен
- [ ] База данных создана
- [ ] Миграции выполнены (`npm run db:migrate`)
- [ ] Тестовые данные удалены (если были)
- [ ] Резервное копирование настроено
- [ ] Индексы созданы для часто используемых полей

### 3. Безопасность

- [ ] Все API endpoints защищены проверкой сессии
- [ ] Админ endpoints проверяют роль `admin`
- [ ] Dev endpoints отключены (`/api/auth/dev` возвращает 403)
- [ ] HTTPS настроен (SSL сертификаты)
- [ ] CORS настроен для production домена
- [ ] `SESSION_SECRET` не используется дефолтное значение

### 4. Производительность

- [ ] Статистика оптимизирована (только `usageDays`)
- [ ] Polling настроен вместо хуков с `refetchInterval`
- [ ] QueryClient настроен с оптимальными `defaultOptions`
- [ ] Двойной polling удален из компонентов
- [ ] Неиспользуемый код удален

### 5. Мониторинг и логирование

- [ ] Логирование настроено (PM2 или другой процесс-менеджер)
- [ ] Мониторинг ошибок настроен (опционально)
- [ ] Мониторинг производительности настроен (опционально)

### 6. Админ-панель

- [ ] Админ создан (`npm run make-admin`)
- [ ] Доступ к `/admin` проверен
- [ ] Страница управления сервером доступна (`/admin/server`)
- [ ] Режим обслуживания протестирован

### 7. Telegram бот

- [ ] Либо **long polling** (процесс `telegram-bot-polling` в PM2), либо **webhook** зарегистрирован на production URL
- [ ] Бот отвечает на команды
- [ ] Mini App URL настроен в BotFather

### 8. Тестирование

- [ ] Все основные функции протестированы:
  - [ ] Вход в систему
  - [ ] Просмотр склада
  - [ ] Создание продажи
  - [ ] Управление долгами
  - [ ] Генерация постов
  - [ ] Просмотр отчётов
- [ ] Ошибки обрабатываются корректно
- [ ] Error Boundary работает

## Процесс обновления

### Перед обновлением

1. [ ] Включить режим обслуживания через `/admin/server`
2. [ ] Дождаться завершения текущих запросов (1-2 минуты)
3. [ ] Создать резервную копию базы данных

### Обновление

1. [ ] Остановить приложение (PM2: `pm2 stop app`)
2. [ ] Обновить код (`git pull` или загрузка новых файлов)
3. [ ] Установить зависимости (`npm install`)
4. [ ] Выполнить миграции БД (`npm run db:migrate`)
5. [ ] Собрать приложение (`npm run build`)
6. [ ] Запустить приложение (`pm2 start app` или `npm start`)

### После обновления

1. [ ] Проверить работу приложения
2. [ ] Выключить режим обслуживания через `/admin/server`
3. [ ] Проверить логи на наличие ошибок
4. [ ] Протестировать основные функции

## Откат (Rollback)

Если обновление прошло неудачно:

1. [ ] Включить режим обслуживания
2. [ ] Остановить приложение
3. [ ] Восстановить предыдущую версию кода
4. [ ] Восстановить базу данных из резервной копии (если нужно)
5. [ ] Запустить предыдущую версию
6. [ ] Выключить режим обслуживания

## Мониторинг после запуска

### Первые 24 часа

- [ ] Проверять логи каждые 2-4 часа
- [ ] Мониторить использование ресурсов (CPU, RAM, диск)
- [ ] Проверять количество активных пользователей
- [ ] Отслеживать ошибки в логах

### Регулярный мониторинг

- [ ] Ежедневная проверка логов
- [ ] Еженедельная проверка резервных копий БД
- [ ] Ежемесячная проверка обновлений зависимостей
- [ ] Мониторинг использования дискового пространства
